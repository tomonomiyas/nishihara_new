ガイドラインに沿った具体的な開発フローをまとめると、以下のようになります。

1. 開発中：yarn dev
   まずは開発サーバーを立ち上げて、ブラウザで確認しながらコードを書きます。

この時点では、コードの書き方が少し乱れていても（インデントがずれていても）問題ありません。
yarn dev はコードを変更するたびにリアルタイムでブラウザを更新してくれます。

2. 区切りがついたら：yarn format
   一つのパーツが完成した、あるいは作業を中断してコミットする前などに実行します。

これで「Lint」と「Prettier」が走り、コードがガイドライン通りのきれいな状態に自動修正されます。
ポイント: VSCodeの設定（保存時に自動フォーマット）をしていれば、この作業は意識しなくても勝手に終わっていることが多いですが、念のため実行するのが確実です。

3. 納品・公開前：yarn build
   最後に、本番用のファイル（distフォルダ）を作るために実行します。

このコマンドを実行すると、内部的に yarn format も自動で実行されます。
なので、最悪 yarn format を忘れていても、ビルドさえ通れば納品物はきれいな状態になります。

まとめ：理想的なルーチン
yarn dev を実行して作業開始
コードを書く（ガシガシ書く）
yarn format もしくは yarn \_lint でチェック（エラーがないか確認）
Gitにコミット
全て完了したら yarn build して、distフォルダの中身をサーバーにアップ

補足： もしビルド後に「あれ、表示が崩れてる？」となるのが不安な場合は、ビルドの後に以下のコマンドを叩くと、本番用ファイルの状態をローカルで確認できます。

**Git の実行手順**

1. 作業開始前（最新の状態にする）
   他の人の変更を取り込み、最新の状態から作業を始めます。

bash
git checkout main
git pull origin main

2. 作業用ブランチの作成
   作業内容に合わせた名前でブランチを作ります。

形式: 種類/作業内容（例: feature/add-header, fix/drawer-bug）
bash
git checkout -b feature/作業名

3. コーディングと保存（add & commit）
   作業をして、区切りがついたら保存します。 ※コミット前に yarn format を実行してコードを整えるのがルールです。

bash

# 1. コードを整える

yarn format

# 2. 変更をステージング

git add .

# 3. コミット（メッセージの形式を守る）

# 形式: "種類: 内容" (例: "add: ヘッダー追加")

git commit -m "add: 作業内容"

4. サーバーに送る（push）
   自分のパソコンにあるブランチを、GitHub（origin）に送ります。

bash
git push origin feature/作業名

5. プルリクエスト (PR) の作成
   GitHub上の画面で PR を作成します。

確認必須: yarn format を実行済みか、ビルドエラーがないか。
レビュワーにチェックしてもらい、OK（Approve）が出たら自分で Merge ボタンを押して main に取り込みます。

6. 後片付け
   マージが終わったら、使い終わったブランチを消して最新の main に戻ります。

bash
git checkout main
git pull origin main
git branch -d feature/作業名
コミットメッセージの主な種類
迷ったら以下を参考にしてください。

add: 新しい機能やファイルを追加したとき
fix: バグや表示崩れを直したとき
update: 既存のものを調整・改善したとき
style: 見た目の微調整（SCSSの修正など）したとき
refactor: コードの中身を整理したとき（見た目は変わらない）
